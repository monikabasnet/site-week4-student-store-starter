🏗️ Architecture: MVC Pattern

Model-View-Controller (Backend)

The project follows a modified MVC architecture pattern, separating concerns into Routes, Controllers, and Models for clean, maintainable code.

📍 Routes

  • Define API endpoints
  • Map URLs to controllers
  • Handle HTTP methods
  • No business logic

🎮 Controllers

  • Handle requests/responses
  • Validate input data
  • Business logic
  • Error handling
  • Format responses

💾 Models

  • Database operations
  • Prisma queries
  • CRUD methods
  • Return raw data

Why This Architecture?

  • Separation of Concerns: Each layer has a specific responsibility
  • Maintainability: Easy to find and fix bugs
  • Scalability: Can add features without breaking existing code
  • Testability: Each component can be tested independently
  • Team Collaboration: Multiple developers can work on different layers

🔄 Request Flow: How Everything Connects

Example: GET /products/1

1️⃣ Client Request
GET http://localhost:3000/products/1
⬇️
2️⃣ Server (server.js)
Receives request, routes to /products
⬇️
3️⃣ Router (productRoutes.js)
Matches /:id pattern → calls controller
⬇️
4️⃣ Controller (productController.js)
Validates ID, calls model
⬇️
5️⃣ Model (product.js)
Executes Prisma query
⬇️
6️⃣ Database (PostgreSQL)
Returns product data
⬆️
7️⃣ Response
JSON data sent back to client

Step-by-Step Breakdown

  1. Client sends request: Browser or Postman sends HTTP request
  2. Express receives: server.js receives and processes with middleware (CORS, JSON parser)
  3. Router matches: Routes file matches URL pattern to controller function
  4. Controller processes: Validates input, handles business logic, error handling
  5. Model queries: Makes database query using Prisma ORM
  6. Database responds: PostgreSQL returns data
  7. Response sent: Data flows back up through model → controller → route → client

💻 Code Example: Complete Flow

1. Route Definition (productRoutes.js)

const express = require('express'); const router = express.Router(); const productController = require('../controllers/productController'); // Map GET /:id to controller function router.get('/:id', productController.getProductById); module.exports = router;

2. Controller Logic (productController.js)

const Product = require('../models/product'); const getProductById = async (req, res) => { try { // Call the model to fetch data const product = await Product.findById(req.params.id); // Send successful response res.status(200).json(product); } catch (error) { // Handle errors res.status(500).json({ error: 'Failed to fetch product' }); } } module.exports = { getProductById };

3. Model Database Query (product.js)

const { PrismaClient } = require('@prisma/client'); const prisma = new PrismaClient(); class Product { static async findById(id) { // Prisma query to database return await prisma.product.findUnique({ where: { id: parseInt(id) } }); } } module.exports = Product;

🗄️ Database Schema

Three Main Tables with Relationships

📦 Product Table

id Integer (Primary Key, Auto-increment)
name String - Product name
description String - Product details
price Decimal - Product price
image_url String - Image link
category String - Product category
order_items Relation[] - Connected OrderItems

🛒 Order Table

order_id Integer (Primary Key, Auto-increment)
customer_id Integer - Customer identifier
total_price Decimal - Total order cost
status String - Order status (pending, completed)
created_at DateTime - Timestamp (auto)
order_items Relation[] - Connected OrderItems

📋 OrderItem Table (Join Table)

order_item_id Integer (Primary Key, Auto-increment)
order_id Integer (Foreign Key → Order)
product_id Integer (Foreign Key → Product)
quantity Integer - Number of items
price Decimal - Price snapshot
order Relation - Connected Order
product Relation - Connected Product

🔗 Table Relationships

Product
⬅️ Many-to-Many ➡️
Order
⬆️
OrderItem
(Junction Table)

Relationship Explanation

  • One Order can have many OrderItems (One-to-Many)
  • One Product can be in many OrderItems (One-to-Many)
  • OrderItem connects Products and Orders (Many-to-Many through junction)
  • Cascade Delete: When an Order is deleted, all its OrderItems are automatically deleted
  • Cascade Delete: When a Product is deleted, all OrderItems referencing it are deleted

📝 Prisma Schema Definition

The schema.prisma file defines the database structure. Prisma uses this to generate the database tables and the Prisma Client for querying.

// Database connection configuration datasource db { provider = "postgresql" url = env("DATABASE_URL") } generator client { provider = "prisma-client-js" } // Product model model Product { id Int @id @default(autoincrement()) name String description String price Decimal image_url String category String order_items OrderItem[] // Relationship } // Order model model Order { order_id Int @id @default(autoincrement()) customer_id Int total_price Decimal status String created_at DateTime @default(now()) order_items OrderItem[] // Relationship } // OrderItem model (junction table) model OrderItem { order_item_id Int @id @default(autoincrement()) order_id Int product_id Int quantity Int price Decimal // Relationships with cascade delete order Order @relation(fields: [order_id], references: [order_id], onDelete: Cascade) product Product @relation(fields: [product_id], references: [id], onDelete: Cascade) }

⚡ Advanced Features

1. Query Filtering & Sorting

Products can be filtered by category and sorted by price or name:

GET /products?category=clothing&sort=price
// Model handles dynamic filtering static async findAll(filters = {}) { const { category, sort } = filters; const queryOptions = {}; // Add category filter if provided if (category) { queryOptions.where = { category: category }; } // Add sorting if provided if (sort === 'price' || sort === 'name') { queryOptions.orderBy = { [sort]: 'asc' }; } return await prisma.product.findMany(queryOptions); }

2. Database Transactions

Problem: When creating an order, we need to create the order AND its items. If one fails, both should fail.

Solution: Prisma transactions ensure data consistency.

// Create order with items atomically static async createWithItems(orderData) { return await prisma.order.create({ data: { customer_id: orderData.customer_id, total_price: orderData.total_price, status: orderData.status, order_items: { create: orderData.items // Create all items in one transaction } }, include: { order_items: true // Return with items included } }); }

3. Cascade Deletes

Automatic Cleanup: When you delete an Order or Product, all related OrderItems are automatically deleted thanks to onDelete: Cascade in the schema.

This prevents orphaned records and maintains database integrity.

🔧 Express Middleware

What is Middleware?

Functions that execute BEFORE your route handlers. They have access to the request and response objects and can modify them.

// server.js const express = require('express'); const cors = require('cors'); const app = express(); // Middleware 1: Parse JSON request bodies app.use(express.json()); // Middleware 2: Enable CORS (Cross-Origin Resource Sharing) app.use(cors()); // Middleware 3: Mount routes app.use('/products', productRoutes); app.use('/orders', orderRoutes); app.use('/order-items', orderItemRoutes);

Middleware Functions

Middleware Purpose Why It's Needed
express.json() Parse JSON request bodies Converts JSON strings to JavaScript objects so you can access req.body
cors() Enable Cross-Origin requests Allows frontend (port 3001) to call backend API (port 3000)
app.use('/path', router) Mount route modules Organizes routes into separate files by resource

Request Processing Order

  1. Request comes in
  2. express.json() parses body
  3. cors() adds CORS headers
  4. Router matches the path
  5. Controller function executes
  6. Response sent back

⚠️ Error Handling

Try-Catch Pattern

Every controller function uses try-catch to handle errors gracefully and send appropriate HTTP status codes.

const getProductById = async (req, res) => { try { // Attempt to fetch product const product = await Product.findById(req.params.id); // Success: send 200 status res.status(200).json(product); } catch (error) { // Error occurred: log it and send 500 status console.error('Error fetching product:', error); res.status(500).json({ error: 'Failed to fetch product' }); } };

HTTP Status Codes Used

Code Meaning When Used
200 OK Successful GET, PUT, DELETE
201 Created Successful POST (resource created)
400 Bad Request Invalid input data
404 Not Found Resource doesn't exist
500 Server Error Database error or unexpected issue

Why Error Handling Matters

  • User Experience: Meaningful error messages instead of crashes
  • Debugging: console.error logs help track down issues
  • API Contract: Consistent error format for frontend
  • Security: Don't expose sensitive error details

💪 Challenges & Solutions

Challenge 1: Database Relationships

Problem: Understanding how to connect Orders, Products, and OrderItems

Solution: Implemented a junction table (OrderItem) to create a many-to-many relationship with proper foreign keys and cascade deletes

Challenge 2: Transaction Handling

Problem: Creating an order with multiple items atomically (all or nothing)

Solution: Used Prisma's nested create feature to create Order and OrderItems in a single transaction

Challenge 3: Code Organization

Problem: Keeping code maintainable as the project grows

Solution: Implemented MVC architecture with clear separation between routes, controllers, and models

Challenge 4: Cascade Deletes

Problem: Ensuring orphaned OrderItems don't remain when Orders or Products are deleted

Solution: Configured onDelete: Cascade in Prisma schema for automatic cleanup
